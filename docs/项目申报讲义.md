# 开场（第一页）

（不要念标题）

各位老师好，我们团队申报的项目，是一个基于大语言模型的 AI Native 事务管理系统。

该项目最核心的一点是：

让软件“理解用户”，而不是让用户去适应软件。

------

# 第二页（问题分析）

（不要逐条念）

我们团队之所以决定做这个项目是因为我们发现，现在大多数记账、日程、提醒类APP，其实还是传统表单式交互。

如右图是一款记账APP，当我们记账时，往往需要：

打开应用、
进入页面、
选择分类、
输入金额、
填写备注、
最后再保存。

如果有多个账目需要记录就要重复点很多次。

虽说微信、支付宝已经支持自动记账，但它只能识别平台内支付。

它并不能真正理解用户说的话。

比如一句：

“今天午饭35元，明天下午3点开会”

传统系统是无法一次完成：

记账 + 日程创建

这两个事务的。

所以我们就在想：

能不能让用户直接说一句话，
剩下全部交给AI完成。

这就是我们整个项目的出发点。

------

# 第三页（系统架构）

（这里不要讲太细）

整个系统我们采用的是：

“单LLM + 模块化Prompt Workflow”架构。

系统主要分成五层，如图所示

这里最核心的是AI事务解析引擎。

因为我们真正研究的重点，不是“调用API”。

而是：

怎么让大模型稳定完成事务理解。

所以我们把：

事务拆分、
意图识别、
结构化提取、
时间解析、
输出校验

全部拆成独立模块。

而这样做的好处是：

稳定性更高，
更容易维护，
也更适合工程落地。

------

# 第四页（核心页）

（这一页决定答辩分数）

这页是整个项目最核心的部分。

这里展示的是：

大模型是怎么一步一步完成事务处理的。

我们以一句真实输入为例：

“今天午饭35元，明天下午3点开会”

系统首先会做：

多意图拆分。

因为一句话里面其实包含两个事务。

然后系统会识别：

第一个是消费记账，
第二个是日程创建。

接着进入 Prompt Workflow。

这里我们会利用：

Few-shot Prompt、
JSON Mode、
Schema约束

让模型稳定输出结构化JSON。

之后再进入：

混合式时间解析。

因为像：

“明天下午3点”
这种时间其实是模糊时间。

所以我们采用：

LLM语义理解 + 规则引擎

共同完成时间转换。

最后系统还会经过：

JSON修复、
Schema校验、
默认值补全、
自动Retry

这些工程化保障。

最终才会真正写入数据库。

所以我们这个项目其实重点研究的是：

# “如何让LLM稳定完成事务执行”

而不仅仅是简单调用模型。

（这一句一定要说）

# 第六页（创新点）

（不要逐条念）

我们项目目前主要有三个创新点。

第一个是：

模块化 Prompt Workflow。

相比传统单Prompt方式，
我们把整个LLM流程拆成多个模块。

第二个是：

结构化输出稳定性保障。

因为大模型经常会出现：

格式错误、
JSON不完整、
输出漂移。

所以我们加入了：

自动修复和多层校验机制。

第三个是：

混合式时间解析。

因为纯LLM时间推理并不稳定。

所以我们结合规则引擎，
提高时间解析准确率。
